The 5 attacks on x402, and how PayGate fixes each one

In May 2026, researchers published a paper identifying 5 concrete attacks on the x402 agentic payment protocol. PayGate is designed against all 5. Here's the full breakdown.

01

Grant-before-settle

The server runs the work BEFORE the on-chain payment is confirmed. A client receives 200 without paying.
❌ Vanilla x402
// 1. server returns 402 with payment reqs
// 2. client retries with payment
// 3. server runs handler IMMEDIATELY
// 4. server calls /settle
// ← race: client cancels the payment tx
//    but server already responded 200
✓ PayGate
// 1. server returns 402 with payment reqs
// 2. client retries with payment
// 3. server calls /settle and AWAITS
//    the on-chain receipt
// 4. server runs handler ONLY after
//    receipt is confirmed
// → no race; no unpaid work
02

Missing resource-identifier binding

A client pays for resource A but the server hands back resource B. No on-chain link between payment and resource.
❌ Vanilla x402
// 402 response is generic
{
  "accepts": [{
    "scheme": "exact",
    "maxAmountRequired": "100"
  }]
}
// No link to which endpoint,
// which resource, which params
// → easy to substitute
✓ PayGate
// PayGate's 402 includes the
// resource path in payment reqs
{
  "accepts": [{
    "scheme": "exact",
    "resource": "/agents/sentiment",
    "maxAmountRequired": "10000"
  }]
}
// resource is bound into the
// EIP-3009 authorization
// → can't be substituted
03

Fire-and-forget settlement

The server claims it settled but the on-chain settlement actually fails or is never made. The service runs anyway.
❌ Vanilla x402
// handler runs first
const result = await handler(input);
// settle in background, fire-and-forget
fetch(facilitator + "/settle", { ... });
// client already got 200
// if /settle fails, no one notices
res.json(result);
✓ PayGate
// settle SYNCHRONOUSLY
const r = await fetch(facilitator + "/settle", { ... });
if (!r.ok) throw new SettlementError();
// await on-chain receipt
const receipt = await publicClient
  .waitForTransactionReceipt({ hash: r.tx });
// only NOW run the handler
const result = await handler(input);
res.json(result);
04

Missing Cache-Control headers

A 402 response gets cached by an intermediate proxy. The client never sees the payment requirements and the service runs once with no payment.
❌ Vanilla x402
// 402 response sent
res.status(402).json({
  accepts: [...]
});
// no Cache-Control header
// → browser/proxy may cache
// → second call gets stale
//    "no payment needed" response
✓ PayGate
res.status(402);
res.setHeader("Cache-Control",
              "no-store");
res.setHeader("Pragma", "no-cache");
res.json({
  accepts: [...]
});
// no-store is also on the
// 200 response — every response
// is dynamic
05

Replay

A client reuses a single payment authorization to call the endpoint multiple times. One payment → many services.
❌ Vanilla x402
// no server-side tracking
// of used authorizations
// EIP-3009 nonce is single-use
// per USDC contract, but
// no per-resource tracking
// → if multiple endpoints
//   accept the same payment
//   scheme, replays are possible
✓ PayGate
// 3 layers of defense:
// 1. EIP-3009 nonce (USDC contract)
// 2. PayGate SpendingPolicy:
//    epoch counter increments
//    on every recordSpend
// 3. Per-call limit: even if
//    the nonce is reused, the
//    per-call cap blocks it
// → no successful replay

Beyond the 5 attacks

PayGate also adds three primitives that x402 itself doesn't have:

  • ERC-8004 identity — every PayGate agent links to a portable on-chain handle on Base
  • On-chain spending policy — per-call cap, per-epoch cap, allowlist, enforced by the contract (not the SDK)
  • Kill switch — one transaction pauses the agent. No admin key, no race condition.
