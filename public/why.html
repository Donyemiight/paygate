Why PayGate, in 4 scenarios

You're building an x402 agent. You have a wallet. You have a budget. Should you adopt PayGate? Here's what changes in each common scenario.

Scenario 1: Your agent is brand new, no real users yet

You're shipping an LLM-backed service that charges per call. You don't have paying customers yet.

❌ Without PayGate

You set up an x402 endpoint. It works. You forget about it. Six months later you have 50 paying customers and your agent's wallet has $5,000 in it. A prompt-injection attack or a bug in your agent code lets an attacker drain it. You notice 12 hours later. The wallet is empty.

✓ With PayGate

You add PayGate at launch. The agent's wallet has a $50/day cap by default. Even if the agent is compromised, the attacker can only drain $50. You can pause the agent from your phone in 30 seconds, before the epoch resets.

Scenario 2: You have an existing x402 service with paying users

Your agent is making money. You have a reputation to protect.

❌ Without PayGate

You're the only thing standing between the agent and a drained wallet. Your monitoring catches the attack, eventually. But the damage is done — your brand takes a hit, and you're explaining to users why their payments went to an attacker.

✓ With PayGate

Your customers see that your agent has a portable ERC-8004 identity with a verified policy. They trust you more. If your agent ever does go rogue, you pause it in 1 transaction, and your customers see the kill-switch event on BaseScan. Transparency is your moat.

Scenario 3: You're building a multi-agent system

Agent A calls Agent B calls Agent C, all paying each other in USDC.

❌ Without PayGate

Each agent has its own wallet. There's no way to set a per-epoch cap on the whole system. If one agent gets stuck in a loop, it can call itself 10,000 times and burn through your budget. You have to manually monitor and kill the bad agent.

✓ With PayGate

Each agent has its own PayGate binding with its own cap. The system has a total budget enforced at the per-agent level. You can pause any single agent without affecting the others. Reputation flows between agents via ERC-8004 — Agent C can see Agent B's track record before paying it.

Scenario 4: You want to monetize your agent

You have an agent. You want to charge for it. You want users to find it, trust it, and pay it.

❌ Without PayGate

You set up an x402 endpoint and a SKILL.md. You post it on X. Users find it. They have no way to verify it's really you. There's no reputation. There's no recourse if the agent is buggy. You're just a URL.

✓ With PayGate

Your agent has an ERC-8004 identity, a PayGate binding, and a visible on-chain policy. Other agents (and their owners) can verify you before paying you. After every successful call, your reputation goes up. You're a discoverable, trustworthy participant in the agent economy — not just a URL.

The 3-line integration

If you've read this far, here's the cost of adopting PayGate:

import { wrap } from "@paygate/sdk";
app.post("/api", wrap(cfg, async (input) => myAgent(input), { priceUSDC: 10000n }));
// that's it. you're done.

What if I'm wrong?

The cost of being wrong (adopting PayGate and not needing it) is sub-cent per call. The cost of being wrong the other way (not adopting PayGate and needing it) is your entire agent wallet. Asymmetric.
